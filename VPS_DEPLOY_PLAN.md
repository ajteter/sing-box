## Current Assessment

The VPS is currently using the official `install.sh` deployment path, not this repository's multi-file deployment layout.

This means:

- The running service is valid, but it is not managed by this repo's `sing-box.sh`.
- `/etc/sing-box` currently has only one `config.json`, while this repo expects a generated `conf/` directory plus `list` and `subscribe/`.
- If this repo script is run directly later, it will likely replace the current `/etc/sing-box` layout and override the current service behavior.
- The current public port `8080` is already being scanned. Recent logs show repeated invalid Shadowsocks probes from `185.91.127.85`.

## Goal

Migrate the VPS from the current single-inbound official install to the repository-managed deployment, with a more suitable protocol set and a reproducible config flow.

Recommended first-stage target:

- Primary protocol: `VLESS + Reality`
- Backup protocol: `Hysteria2`
- Optional second backup: `TUIC`
- Do not enable all protocols in the first round
- Do not enable Argo or WebSocket/CDN in the first round unless there is a clear need

Recommended protocol choice for first deployment:

```text
CHOOSE_PROTOCOLS='bcd'
```

Meaning:

- `b`: VLESS + Reality
- `c`: Hysteria2
- `d`: TUIC

## Migration Strategy

### Phase 1: Backup Current State

Before switching to the repo-managed layout, back up the current official installation.

Commands to run on the VPS:

```bash
mkdir -p /root/sing-box-backup-2026-04-20
cp -a /etc/sing-box /root/sing-box-backup-2026-04-20/etc-sing-box
cp -a /lib/systemd/system/sing-box.service /root/sing-box-backup-2026-04-20/sing-box.service
systemctl status sing-box --no-pager > /root/sing-box-backup-2026-04-20/status.txt
journalctl -u sing-box -n 100 --no-pager > /root/sing-box-backup-2026-04-20/journal.txt
```

### Phase 2: Prepare Repo-Driven Parameters

Prepare a dedicated config file based on this repository's `config.conf`.

Recommended first-pass values:

```bash
LANGUAGE='c'
CHOOSE_PROTOCOLS='bcd'
START_PORT='30000'
PORT_NGINX=''
SERVER_IP='64.83.43.146'
CDN=''
UUID_CONFIRM='<generate-a-new-uuid>'
SUBSCRIBE='true'
ARGO='false'
VMESS_HOST_DOMAIN=''
VLESS_HOST_DOMAIN=''
ARGO_DOMAIN=''
ARGO_AUTH=''
PORT_HOPPING_RANGE='50000:51000'
REALITY_PRIVATE=''
NODE_NAME_CONFIRM='my-sing-box'
```

Notes:

- `PORT_NGINX` stays empty because the first deployment does not use WebSocket subscription over nginx.
- `REALITY_PRIVATE` can be left empty and generated by the script.
- `UUID_CONFIRM` should be newly generated rather than reusing the current Shadowsocks password.
- `SUBSCRIBE='true'` is useful because this repo can export client configs and subscription files.

UUID generation example:

```bash
cat /proc/sys/kernel/random/uuid
```

### Phase 3: Stop Mixing Two Management Models

Do not manually maintain both:

- official `/etc/sing-box/config.json`
- repo-generated `/etc/sing-box/conf/*.json`

Only one model should own the service.

Recommended approach:

- Keep the official install only as a temporary running state.
- After backup, switch fully to the repo-managed model.

### Phase 4: Deploy With This Repo

Run this repository's script using the config file mode.

Suggested command:

```bash
bash <(wget -qO- https://raw.githubusercontent.com/fscarmen/sing-box/main/sing-box.sh) -f config.conf
```

After deployment, the expected layout should look like:

```text
/etc/sing-box/conf/
/etc/sing-box/list
/etc/sing-box/subscribe/
/etc/sing-box/sing-box
```

The expected service command after repo takeover should look like:

```text
/etc/sing-box/sing-box run -C /etc/sing-box/conf
```

## Validation Checklist

After migration, verify all of the following:

```bash
systemctl status sing-box --no-pager
ss -lntup | grep sing-box
ls -la /etc/sing-box
find /etc/sing-box/conf -maxdepth 1 -type f | sort
cat /etc/sing-box/list
ls -la /etc/sing-box/subscribe
journalctl -u sing-box -n 100 --no-pager
```

Expected checks:

- `sing-box` stays `active (running)`
- ports for `Reality`, `Hysteria2`, and `TUIC` are listening
- `/etc/sing-box/conf/` exists
- `/etc/sing-box/list` contains exported node information
- `/etc/sing-box/subscribe/` contains generated client profiles or subscription files

## Security Notes

- The current port `8080` is being probed on the public internet.
- After migration, avoid exposing only a single easy-to-scan Shadowsocks port.
- Prefer `Reality` as the primary entry point.
- If `Hysteria2` is enabled with port hopping, confirm firewall or NAT rules were created correctly.
- Do not store SSH passwords or long-term secrets in tracked files.

## Recommended Next Action

1. Back up the current official install on the VPS.

3. Run the repo-managed installation.
4. Validate ports, service health, and exported client files.
5. If stable, discard the old single-file config model and manage everything through this repo script only.
